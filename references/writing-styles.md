# Writing Styles Reference

> 文章写作风格库。write skill 根据内容类型自动选择，也支持用户手动指定。

---

## Style A: 技术教程

**适用场景：** 教程、指南、入门文章、实战操作

**代表文章：** 「5分钟用 Docker 部署你的第一个 Web 应用」

**核心特征：**
- Obsidian callouts（> [!info]、> [!tip]、> [!warning]）
- 完整可运行的代码示例（含注释、类型标注、错误处理）
- 参数对比表格
- 段落长度：100-150 字
- 配图节奏：每 400-600 字一张（Gemini 生成图为主）
- 语气：专业但不枯燥，适度使用第一人称
- 结构：线性递进，每节解决一个问题

**开头模式：** 痛点 → 方案 → 本文价值

**每节结构：**
```
## 章节标题

[问题/需求描述]

> [!info] 关键概念
> 解释

[代码示例 + 解释]

[代码示例 + 解释]

[小结或过渡]
```

**结尾：** 具体的下一步操作（一条命令或一个动作）

**截图占位符：** 每当引用工具的**安装文档 / GitHub README / CLI 运行结果页**时，紧随引用段发射一个 `<!-- SCREENSHOT: url [#selector] -->`。触发规则见 `skills/write/SKILL.md` §3f『何时发射 SCREENSHOT 占位符』。

## Style B: 经验分享 / 口语化

**适用场景：** 工具分享、技巧清单、个人经验、"N个你不知道的..."

**代表文章：** 「分享10个你可能不知道的Claude Code隐藏命令」（数字生命卡兹克）

**核心特征：**
- **极短段落** — 1-2 句一段，绝不超过 3 句。像发微信消息
- **口语化语气** — "这玩意儿"、"贼好用"、"说实话"、"真的"
- **第一人称为主** — 大量使用"我"，分享个人真实经历
- **中文数字编号** — **一. 二. 三.**（加粗），不用 1. 2. 3.
- **高频截图** — 每 2-4 段一张截图
- **不用 callouts** — 太正式
- **不用参数对比表格** — 用口语描述替代
- **"。。。" 拖尾** — 表达感叹和留白
- **重复强调词** — "真的"、"太…了"、"属于…的那种"

**开头模式：**
```
最近发现一个很有意思的现象。

[具体场景/对话/事件]

[引出主题："所以我今天把...整理了一下"]

话不多说，我们开始。
```

**每节结构：**
```
**一. 功能名称**

[这是什么 — 1-2 句]

[以前的痛点场景]

[截图]

[怎么用]

[截图]

[个人感受："贼好用，用了就回不去了"]
```

**结尾：** "写在最后" + 情绪升华 + 行动号召 + 签名

**截图占位符：** Style B 本来就是**截图密集型**（每 2-4 段一张），**每个介绍的工具/技巧都要配 SCREENSHOT**：官方页、X 原帖、产品截图都走 `<!-- SCREENSHOT: url [#selector] -->`。触发规则见 `skills/write/SKILL.md` §3f。

## Style C: 深度长文

**适用场景：** 技术原理、架构设计、源码分析、RFC 解读

**代表文章：** 「Go GC 调优实战：从 200ms 停顿降到 5ms 的全过程」

**核心特征：**
- 段落较长（150-200 字），论述充分
- 大量原理图和架构图（Mermaid → 图片占位符）
- 先给结论，再展开论证
- 引用论文/RFC/源码（inline link）
- 配图节奏：每 500-800 字一张
- 代码以源码片段为主，配合行号注释
- 语气：严谨但可读，像在做技术演讲

**开头模式：** 结论先行 → 为什么重要 → 本文结构预览

**每节结构：**
```
## 章节标题

[本节核心结论（1 句话）]

[原理展开 — 问题背景、设计动机]

[架构图/流程图]

[源码关键路径分析]

[性能数据/基准测试]

[本节总结 — 回扣核心结论]
```

**结尾：** 总结要点 + 延伸阅读方向 + 开放问题

## Style D: 评测对比

**适用场景：** 产品对比、框架选型、工具评测、方案选择

**代表文章：** 「Bun vs Deno vs Node.js — 2024 运行时终极对比」

**核心特征：**
- **结构化对比** — 多维度表格是核心元素
- **公平呈现** — 每个选项都说优点和缺点
- **有明确结论** — 不做"各有千秋"的骑墙派
- 大量基准测试数据和截图
- 段落适中（80-120 字）
- 语气：客观但有态度，用数据说话
- 配图：性能图表、功能对比表为主

**开头模式：** 选型困惑场景 → 本文对比维度预览 → "先说结论"

**每节结构：**
```
## 维度：性能/生态/DX/...

[这个维度为什么重要]

| 维度 | 选项 A | 选项 B | 选项 C |
|------|--------|--------|--------|
| 指标1 | 数据 | 数据 | 数据 |
| 指标2 | 数据 | 数据 | 数据 |

[数据解读 — 谁赢了、为什么]

[截图/图表]

> [!tip] 选择建议
> 在...场景下选 A，在...场景下选 B
```

**结尾：**
```
## 最终推荐

| 场景 | 推荐 | 原因 |
|------|------|------|
| 个人项目 | B | ... |
| 企业生产 | A | ... |
| 学习目的 | C | ... |

[一句话总结你的个人选择和理由]
```

**截图占位符：** 评测必须配**真实的基准测试页 / 官方性能报告 / 排行榜**截图，每个对比维度尽量配一张：`<!-- SCREENSHOT: url [#selector] -->`。跑分截图最能服人。触发规则见 `skills/write/SKILL.md` §3f。

## Style E: 资讯快报

**适用场景：** 新版本发布、技术新闻、行业动态、更新日志解读

**代表文章：** 「Claude Code 3.0 更新：5 个最值得关注的新功能」

**核心特征：**
- **快速扫描友好** — 每个要点 2-5 段，读者能快速 skim
- **信息密度高** — 去掉所有废话，只保留事实和评价
- **时效性标注** — 明确说明"什么时候发布的"
- 段落极短（1-3 句）
- 大量链接（原始来源）
- 截图为主（官方截图、Twitter 帖子截图）
- 语气：简洁直接，新闻+轻评论

**开头模式：**
```
[产品/技术] 在 [日期] 发布了 [版本/更新]，带来了 N 个重要变化。

快速过一遍最值得关注的。
```

**每节结构：**
```
### N. 功能名称

[一句话说清是什么]

[为什么重要 — 1-2 句]

[截图]

[官方链接或来源]
```

**结尾：** 个人评价（值不值得升级/关注） + 官方链接汇总

**截图占位符：** 资讯快报对**原始信源截图**要求最高：官方发布公告页、changelog、CEO/PM 的 X 原帖，每个要点都要配一张 `<!-- SCREENSHOT: url [#selector] -->`。这是"真实性锚点"。触发规则见 `skills/write/SKILL.md` §3f。

## Style F: 项目复盘 / Case Study

**适用场景：** 项目总结、踩坑记录、架构演进、迁移复盘

**代表文章：** 「我们如何将 API 响应时间从 2s 降到 50ms」

**核心特征：**
- **叙事驱动** — 像讲故事一样，有时间线、有转折、有结局
- **真实数据** — before/after 对比，具体的数字
- **坦诚踩坑** — 不隐藏错误决策和弯路
- 段落适中（80-150 字）
- 混合使用代码、架构图、数据图表
- 语气：复盘总结式，冷静但有感情
- 时间线结构或问题→尝试→失败→成功结构

**开头模式：**
```
[结果先行 — 我们做到了什么]

[背景 — 之前是什么状况，为什么要改]

[本文结构 — "这篇文章会讲三件事：..."]
```

**每节结构：**
```
## 阶段 N：[做了什么]

[当时的状况和指标]

[我们的方案]

[代码/配置关键变更]

[结果 — 数据对比]

> [!warning] 踩坑
> [这里出了什么问题，怎么解决的]
```

**结尾：**
```
## 复盘总结

**做对了的：**
- ...

**做错了的：**
- ...

**如果重来会怎么做：**
- ...

[关键 lesson learned — 一句话]
```

**截图占位符：** 复盘贵在"有据可查"：**监控面板 before/after、PR 页、issue 页、报错日志页**都配 `<!-- SCREENSHOT: url [#selector] -->`。如果是内部系统截不到公开 URL，允许退回手工图片（不发射 SCREENSHOT 占位符）。触发规则见 `skills/write/SKILL.md` §3f。

## Style G: 观点输出 / 思考

**适用场景：** 技术观点、行业思考、趋势判断、方法论分享

**代表文章：** 「为什么我不再推荐 TypeScript」

**核心特征：**
- **观点鲜明** — 标题就亮明立场，不含糊
- **论证充分** — 每个观点配 2-3 个论据
- **允许争议** — 预设反驳并回应
- 段落中等（100-150 字）
- 少代码，多论述
- 语气：自信但不傲慢，"我认为"而非"显然"
- 配图少，以文字论证为主

**开头模式：**
```
[争议性结论 — 直接抛出]

[这不是标题党 — 简短说明为什么这么想]

[本文结构/论证路径预览]
```

**每节结构：**
```
## 论点 N：[核心主张]

[论据 — 数据/案例/对比]

[论据 — 亲身经历]

[预设反驳："你可能会说..."]

[回应反驳]

[本节结论]
```

**结尾：** 总结立场 + 承认局限性 + 期待讨论

**截图占位符：** 观点文配图最少，但**引用"反方"或"同方"的别家文章/推文时必截**，否则容易被质疑"断章取义"——直接发射 `<!-- SCREENSHOT: url [#selector] -->` 锁证。触发规则见 `skills/write/SKILL.md` §3f。

## Style H: AI 资讯爆料 / 自媒体爆款

**适用场景：** 公众号爆款、AI 产品发布爆料、泄露代号对照、行业对垒叙事、股价反应报道

**代表文章：** 「Claude Opus 4.7 刚刚曝光！Claude Code 一夜重构，7x24 小时替你打工」（AI 科技爆料类公众号风格）

**核心特征：**
- **戏剧性标题** — "刚刚曝光"、"一夜重构"、"突袭"、"一刀直捅"、"神仙打架"、"降智"
- **【导读】开篇** — 加粗、1-3 句话先把爆点全抛出
- **超短段落** — 绝大多数 1-3 句一段，换行即是节奏
- **H2 加粗钩子句** — 每节小标题是悬念/结论，不是中性描述（"睡一觉，它把活都干了" ≠ "功能说明"）
- **口语网感词汇** — "炸了"、"上大分"、"神仙打架"、"土法炼钢"、"一刀直捅"、"硬刚"、"折腾"、"直播"
- **竞争/对手叙事** — 必提竞品（Adobe / Figma / OpenAI / Gamma），股价反应、挑战、威胁是必备钩子
- **爆料 / 泄露对照** — 泄露代号（KAIROS / capybara-v2）与官方发布一一对应，制造"我早就说了"感
- **证据截图密集** — 每 2-3 段一图：X 原帖截图、官方博客截图、泄露代码/文件截图、股价截图、产品界面 GIF
- **源图直引** — **不重截**，直接复用源文章的图片 URL（`mmbiz.qpic.cn` / `pbs.twimg.com` / `i.imgur.com`）
- **段间过渡口头禅** — "更刺激的是"、"但这还不是最炸的"、"关键词是"、"最狠的一个例子"
- **硬核干货穿插** — 要有真 curl / 事件列表 / 参数表，用比喻软化（"工作说明书"、"云端员工"、"一 PR 一 session"）
- **公众号三板斧收尾** — "点赞、转发、在看一键三连" + "点亮星标" + 签名/订阅号二维码占位

**开头模式：**
```
[戏剧化导语 — 加粗、1-3 句抛爆点]

[产品 / 人物] 刚刚 [动作]。

[时间节点 — "就在今天" / "刚刚" / "不到 X 小时前"]

[一句话事实 + 一张证据截图（官方推文 / 博客头图）]

[本文承诺 — "让我们看看他们端出了什么"]
```

**每节结构：**
```
**H2 加粗钩子句（不是客观标题）**

[一句话结论/爆点]

[展开 1-3 句]

![图片](源站 URL)

[干货 — 代码块 / 参数表 / 事件列表]

[解读 — "这东西最直接的用法是..."]

![图片](源站 URL)

[过渡口头禅 — "但这还不是最炸的" / "最狠的一个例子"]
```

**竞争叙事块（几乎必配）：**
```
**对手告急：[产品] 直捅 [竞品] 心脏**

[爆料或动作]

[竞品名单：Adobe、Figma、Gamma、Wix...]

[股价/社交反应：股价跌 X%、网友评论截图]

![图片](源站 URL)
```

**泄露对照块（有泄露素材时必用）：**
```
**两周前泄露的「[代号]」，就是它？**

[提一下先前的泄露事件]

对照一下今天的产品页，你会发现几乎是一一对应。

- 「[泄露描述 1]」→ [正式功能 1]
- 「[泄露描述 2]」→ [正式功能 2]
- 「[泄露描述 3]」→ [正式功能 3]

[一句话收束：这大概是过去一年最干净的一次路线图重合。]
```

**结尾：**
```
参考资料：

[官方博客 URL]
[X 原帖 URL]
[爆料媒体 URL（The Information / The Verge）]

[订阅号签名句 — "秒追 ASI" 这类]

**⭐点赞、转发、在看一键三连⭐**

**点亮星标，锁定 [账号名] 极速推送！**

![图片](公众号二维码)
```

**强制硬约束（write 阶段校验，见 write/SKILL.md）：**
1. 至少 2 张来自 `_evidence.json` 的源站图片（直引远端 URL，不走本地 CDN）
2. 至少 1 个竞争对手名或股价/反应数据
3. 每个 H2 标题必须是钩子句（含感叹号 / 动词 / 修辞），不得是中性描述
4. 必须有【导读】加粗开篇
5. `materials.md` 缺失 / 无公开源 URL → block，提示用户补材料

**禁止（Style H 特有）：**
- 不用 Obsidian callouts（太克制）
- 不用代码块以外的 `> ` blockquote（太书面）
- 不用"综上所述"、"总的来说"等总结句（自媒体不做学术总结）
- 不用主编式客观语气（"值得注意的是"、"从技术角度看"）

## 风格自动判断规则

| 内容信号 | 推荐风格 |
|---------|---------|
| "教程"、"指南"、"入门"、"实战"、"部署" | Style A（技术教程） |
| "分享"、"推荐"、"技巧"、"隐藏"、标题含"N个" | Style B（经验分享） |
| "原理"、"源码"、"架构"、"设计"、"底层" | Style C（深度长文） |
| "对比"、"评测"、"vs"、"选型"、"哪个好" | Style D（评测对比） |
| "更新"、"发布"、"新版本"、"changelog" | Style E（资讯快报） |
| "复盘"、"踩坑"、"迁移"、"优化了"、"从X到Y" | Style F（项目复盘） |
| "为什么"、"我认为"、"不推荐"、"应该" | Style G（观点输出） |
| "曝光"、"爆料"、"突袭"、"泄露"、"一夜"、"刚刚"、"硬刚"、"神仙打架"、"对垒"、提及股价/竞品股价、泄露代号对照 | Style H（爆料自媒体） |
| 来自 YouTube 视频转文章 | Style B（经验分享） |
| 默认 | Style A（技术教程） |

**Style H 触发强信号（任一即命中）：**
- 标题含"刚刚"、"曝光"、"一夜"、"突袭"、"闪电"、"泄露"、"爆料"
- 主题涉及 AI 产品发布/挑战/竞争关系（Anthropic vs OpenAI、Figma vs Claude、等）
- 用户提供了 `materials.md` 或明确指定 `--style=H`
- 用户来源链接包含 AI 科技爆料类公众号（`mp.weixin.qq.com` 的 AI 资讯号）

如果不确定，使用 AskQuestion 让用户选择。

## 通用规则（所有风格适用）

1. **YAML frontmatter** — 必须包含 title, date, tags, category, status, description
2. **description** — ≤ 120 字，有意义的摘要
3. **禁止 AI 八股文** — "在当今"、"综上所述"、"让我们一起探索"
4. **禁止营销词** — "赋能"、"颠覆"、"极致"、"一站式"
5. **标题无 emoji** — 标题和章节标题不用 emoji
6. **inline links** — 所有链接使用 inline 格式，不单独列参考资料
7. **图片占位符** — `<!-- IMAGE: name - desc (ratio) -->` + `<!-- PROMPT: ... -->`
8. **个人视角** — 每篇至少 2 处个人经验/观点（即使是 Style E 资讯型）
9. **self-check** — 保存前执行 `references/self-check-rules.md` 全部规则
